
Use Case name:
Application Shimming Detection (sdbinst.exe)

Objective:
- Detect potential persistence or privilege escalation techniques leveraging application shimming through the execution of sdbinst.exe, a known Windows Living-Off-the-Land Binary (LOLBIN).

Environment 
- **SIEM:** Wazuh (self-hosted Ubuntu VM)
- **Endpoint:** Windows 11
- **Logging:** Sysmon + Wazuh agent

MITRE ATT&CK Mapping
- **Technique:** T1546.011 – Application Shimming
- **Tactic:** Persistence, Privilege Escalation

Detection Source
- A high-severity alert (Level 12) was generated by Wazuh, based on Sysmon process creation telemetry (Event ID 1), identifying the execution of `sdbinst.exe.`
- This activity triggered a predefined Wazuh detection rule associated with application shimming, a technique commonly used by adversaries to establish persistence or escalate privileges.

Investigation
	Process Analysis
	- Executable: `C:\Windows\System32\sdbinst.exe`
	- Command line: `sdbinst.exe -m -bg`
	- Executed with **SYSTEM privileges**
	- Parent process: `svchost.exe`
	- Associated service: **Program Compatibility Assistant Service (PcaSvc)**
		- This is a legitimate Windows service responsible for compatibility operations

File Integrity Validation
- Digital signature verified → **Valid Microsoft signature**
- Binary located in expected system directory (`System32`)
- No evidence of tampering

Behavioral Context
- No suspicious parent-child process chain observed
- No abnormal command-line arguments
- Activity likely triggered by system or software compatibility operations (e.g., driver/software update)

Risk Assessment (Consulting Perspective)
- **Initial Risk:** High
    - Application shimming is a known persistence mechanism used by attackers
- **Actual Risk:** Low
    - Execution context aligns with legitimate Windows behaviour
    - No indicators of compromise identified

Conclusion
- The alert was triggered due to the execution of a known LOLBIN (sdbinst.exe) associated with application shimming techniques.
- Following investigation, the activity was determined to be benign, initiated by a legitimate Windows service (PcaSvc), with a verified Microsoft-signed binary and no suspicious behaviour observed.
### **Technical Recommendations**
- Maintain monitoring of `sdbinst.exe` executions as a high-signal detection
- Enhance detection by correlating:
    - Unusual parent processes (e.g., Office → cmd → sdbinst)
    - Unsigned or non-System32 binaries
- Consider refining Wazuh rules to reduce false positives:
    - Lower severity when:
        - Parent process = `PcaSvc`
        - Binary is Microsoft signed
### **Business / Consulting Recommendations**
- No immediate remediation required
- Continue monitoring for persistence techniques using LOLBINs
- Incorporate this detection into broader endpoint threat detection strategy aligned with MITRE ATT&CK


Sysmon intentionally retains high-fidelity telemetry without over-filtering, and false positives are expected at the SIEM layer. In my lab, I validated alerts and identified benign PowerShell execution policy artifacts, demonstrating the need for detection tuning rather than relying solely on default rules
